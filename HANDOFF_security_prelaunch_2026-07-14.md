| 1 | ยกเลิกเวรที่เผยแพร่แล้ว ไม่ยืนยันรหัสผ่านจริงฝั่ง server | **ควรแก้เต็มก่อน Friday** | คนใช้งานจริงพึ่งระบบนี้ตัดสินใจเรื่องเวรงาน — แก้ scope เล็ก งานไม่กี่ชั่วโมง ไม่มีเหตุผลจะไม่แก้ |

## เรื่องที่ 1 — 🔴 ความปลอดภัย (4 จุดย่อย)

### 1a. ยกเลิกเวรที่เผยแพร่แล้ว — รหัสผ่านเช็คแค่ฝั่งจอ ไม่ผูกกับการลบจริง

**อาการ:** กดยกเลิก → ระบบขึ้นกล่องให้พิมพ์รหัสผ่านซ้ำ + เหตุผล → **แต่ขั้นตอนลบจริงไม่ได้เช็คว่าผ่านการยืนยันนี้มาก่อนหรือเปล่า** ใครก็ตามที่มี session hash อยู่แล้ว (เช่นเปิด console) เรียกฟังก์ชันลบตรงๆ ได้เลยโดยข้ามกล่องรหัสผ่านไปเฉยๆ

**โค้ดที่เกี่ยวข้อง:**
- `phxVerifyPassword` — `Phase_Z_B1_Auth.js:220` (แค่ตรวจแล้วคืนค่า true/false ไม่ได้สร้างอะไรไว้ยืนยันภายหลัง)
- `swfCancelPublished` — `Index.html:6148`, `swfAdminCancelPublished` — `Index.html:6249` (เรียก verify ก่อน แต่ไม่ได้ส่งผลอะไรต่อไปให้ขั้นตอนถัดไปเช็ค)
- `phxRemoveAction` — `Phase_Z_B3_Sync.js:184` (ตัวลบจริง เช็คแค่ `_phxVerifyAuth` = session hash ปกติ เหมือน call อื่นๆ ทุกอัน)

**แนวทางแก้ที่แนะนำ:** ให้ `phxVerifyPassword` เมื่อผ่านแล้วออก token สั้นๆ (อายุ ~2-5 นาที เก็บฝั่ง server เช่นใน CacheService ผูกกับ session hash) แล้วให้ `phxRemoveAction` เวอร์ชัน cancel-published **บังคับแนบ token นี้มาด้วย** และเช็คว่ายังไม่หมดอายุ — งานเล็ก ไม่กระทบ flow อื่น

---

### 1b. Session ไม่มีวันหมดอายุ — password hash ใน localStorage คือบัตรผ่านตลอดชีพ

**อาการ:** login แล้ว client เก็บ `siriraj_logged_in_name`/`_hash`/`_role` ไว้ใน localStorage แบบไม่มีวันหมดอายุ ส่งแนบเป็น bearer token ทุกครั้งที่เรียก API ที่ต้องมีสิทธิ์ — **ไม่มีตาราง session ฝั่ง server เลย** ถ้าใครขโมยเครื่อง/ก็อปค่านี้ไปได้ จะเข้าบัญชีได้ตลอดจนกว่าเจ้าของจะเปลี่ยนรหัสผ่านเอง

**โค้ดที่เกี่ยวข้อง:**
- `phxAuthSetSession` — `Index.html:11908` (เขียน 3 คีย์ลง localStorage แบบไม่มี timestamp/TTL)
- `_phxVerifyAuth` — `Phase_Z_B3_Sync.js:270` (ทุก push/pull/remove เช็คแค่ hash ตรง sheet ไม่เช็คเวลาเลย)
- คอลัมน์ `lastSeen` มีอยู่แล้วใน Sheet `PHX_Pharmacists` (col D) — แค่ยังไม่ถูกใช้บังคับ expiry

**แนวทางแก้ — 2 ระดับ ให้ Klui เลือก:**
1. **เบา (ทำได้ใน 3 วัน, แนะนำถ้าจะทำ):** เพิ่มเงื่อนไขใน `_phxVerifyAuth` — ถ้า `lastSeen` ห่างเกิน N วัน (เช่น 30) ให้ปฏิเสธและบังคับ login ใหม่ ผู้ใช้ทั่วไปที่เข้าแอปสม่ำเสมอแทบไม่รู้สึกอะไร แก้จุดเดียว ไม่กระทบ flow login/logout ปกติ
2. **เต็มรูปแบบ (เลื่อนหลัง launch):** สร้างระบบ session table จริง + refresh token + revoke ได้จากฝั่งแอดมิน — งานใหญ่กว่ามาก เสี่ยง regression สูงถ้ารีบทำใน 3 วัน

### 1c. แอดมินแก้ผู้ให้/ผู้รับบน record ที่เผยแพร่แล้ว — ไม่มี audit log เลย

**อาการ:** `_pbEditRecipient` (แก้เอง) และ `_pbAdminOverride` (แอดมินแก้ของคนอื่น) บังคับให้กรอกเหตุผลก่อนทุกครั้ง แต่**ไม่เคยเรียก `phxAuditLog()`** ต่างจาก action อื่นทุกชนิดในระบบที่ log ครบ ร่องรอยเดียวที่เหลือคือ field `_overrideBy/_overrideAt/_overrideReason` ที่ฝังอยู่ใน record — ซึ่งถูกเขียนทับได้เรื่อยๆ ถ้ามีการแก้ซ้ำ

**โค้ดที่เกี่ยวข้อง:**
- `_pbEditRecipient` — `Index.html:3323`
- `_pbAdminOverride` — `Index.html:3400`
- เทียบกับ pattern ที่ถูกต้อง: `phxLogAudit()` ใน `Phase_G_AuditLog.js:97`, เรียกผ่าน client wrapper `phxAuditLog()` ที่ `Index.html:14421`

**แนวทางแก้ที่แนะนำ:** เพิ่ม `phxAuditLog('admin_override_recipient', ...)` ต่อท้ายทั้ง 2 ฟังก์ชัน (before/after state + เหตุผลที่กรอกอยู่แล้ว) — งานเล็กมาก ไม่มี regression risk

### 1d. LINE webhook ไม่ verify ลายเซ็น (HMAC signature)

**อาการ:** มีตัวแปร secret พร้อมใช้ แต่ `doPost` (จุดรับ webhook จาก LINE) ไม่เคยเรียกมาตรวจสอบเลย — ใครก็ยิง POST มาที่ endpoint นี้แล้วอ้างว่ามาจาก LINE ได้ ระบบจะเชื่อและประมวลผลทันที

**โค้ดที่เกี่ยวข้อง:**
- `_phxLineGetSecret()` — `Phase_F2_LINE.js:25` (มีฟังก์ชันอยู่แล้ว แต่ไม่มีใครเรียกในทั้งไฟล์)
- `doPost` — `Phase_F2_LINE.js:216-235` (ไม่ verify header `X-Line-Signature`)

**แนวทางแก้ที่แนะนำ:** คำนวณ HMAC-SHA256 ของ request body ด้วย channel secret แล้วเทียบกับ header `X-Line-Signature` — ถ้าไม่ตรง return 401 ก่อนประมวลผลอะไรทั้งสิ้น งาน scoped ชัดเจน ไม่กระทบส่วนอื่น

## เรื่องที่ 2 — 🟠 "เดือน" มี ID 3 สูตรไม่ตรงกัน — รากของปัญหา "Firebase ดับ"

**อาการ:** คำว่า "เดือน" มี ID **3 สูตรที่ไม่ตรงกันเลย** ในระบบ:
- **Sheet/MONTH_LIST** — `monthId` = timestamp สุ่ม (ผ่าน Schedule_Index)
- **Firebase** — `monthId` = คำนวณจากชื่อเดือนภาษาไทย `"m_"+label.replace(/\s+/g,'_')` — **คำนวณแยกกันคนละที่ 2 จุด** (server เขียน vs client อ่าน) ต้องตรงกันเป๊ะทุกตัวอักษรถึงจะหากันเจอ
- **ปฏิทิน (Calendar sync)** — `month_id` = `"m_"+monthValue` — สูตรที่ 3 อีกแบบ ใน `Phase2C.js`

**ผลกระทบที่เคยเกิดจริง:** label เพี้ยนเล็กน้อย (whitespace จาก Excel, พิมพ์ชื่อเดือนต่างกันนิดเดียว) → Firebase หา key ไม่เจอ → รอ 6 วินาทีแล้ว fallback ไป GAS → ผู้ใช้เห็นเป็น **"Firebase ล่ม"** ทั้งที่จริงๆ แค่ป้ายไม่ตรงกัน — เกิดขึ้นมาแล้วหลายรอบตามที่ Klui เจอเอง

**โค้ดที่เกี่ยวข้อง:**
- `pushToFirebase_` — `code.js:511` (server คำนวณ+เขียน key)
- `fetchMonthData` — `Index.html:7108` (client คำนวณ key เองอีกรอบ แยกจาก server)
- `_p2c_appendSyncRow` — `Phase2C.js:201` (ปฏิทินใช้สูตรที่ 3, sheet `User_Calendar_Sync`)

**แนวทางแก้ — ให้ Klui เลือกระดับ:**
1. **แก้เต็ม (ไม่แนะนำก่อน Friday):** รวมทั้ง 3 สูตรเป็นสูตรเดียว + migrate ข้อมูลเก่า ต้องแตะ `code.js`/`Index.html`/`Phase2C.js` พร้อมกัน — เสี่ยง regression สูงเกินไปในเวลาจำกัด
2. **Mitigation แนะนำก่อน Friday:** ดึงสูตร "label → Firebase key" มารวมเป็นจุดเดียวที่ทั้ง server และ client เรียกใช้ร่วมกัน (ไม่ต้อง copy สูตรมือ 2 ที่เหมือนตอนนี้) — ป้องกันไม่ให้เกิดความเพี้ยน**ใหม่** ระหว่างสัปดาห์เปิดตัว แม้จะไม่แก้ของเก่าที่เพี้ยนอยู่แล้วก็ตาม
3. **ลดความน่ากลัวของอาการ (ทำคู่กับข้อ 2 ได้ ง่ายกว่า):** ตอน fallback ไป GAS ให้ขึ้นข้อความปกติแบบ "กำลังโหลด..." แทนที่จะดูเหมือนระบบพัง — ตรงกับที่ Klui เคยขอเรื่อง tone ไม่ให้ user ตกใจ (ดู memory `feedback_changelog_tone`)

## เรื่องที่ 3 — 🟠 LWW ownership record ไม่เคยถูกบันทึกจริง

**อาการ:** เอกสารออกแบบ `DESIGN_LWW_ownership_v3.44.md` ระบุว่าจะมี "สมุดบันทึกถาวร" เก็บว่าใครถือเวรอยู่ตอนนี้ (`{slotKey, newOwner, recordedBy, at, _visibility}`) แต่ **ในโค้ดจริงไม่เคยเขียนข้อมูลรูปแบบนี้ลงที่ไหนเลย** — `LWW.buildRecords()` (Index.html) คำนวณสดใหม่ทุกครั้งจาก overlay/action record เดิม (give/add/swap) ไม่มีการ persist ผลลัพธ์นี้เลยสักที่

**อัปเดตสำคัญ (ต่างจากตอนพบครั้งแรก):** สาเหตุที่เคยเชื่อมกับ "relay legs" (เวรแบ่งไม้ = multi-owner ที่อาจขัดกับ LWW แบบ single-owner) **หมดความเกี่ยวข้องแล้ว** — relay legs ถูกลบทิ้งไปแล้ว (13 ก.ค.) และระบบที่มาแทน (joint co-ownership) ก็ถูกแทนที่อีกทีด้วย **"โน้ตเวร"** ซึ่งเป็น single-owner ล้วนๆ ไม่มี multi-owner semantics แล้ว — จึงไม่มีความขัดแย้งแบบ "เวรมีทั้งเจ้าของเดียวและหลายเจ้าของพร้อมกัน" ให้กังวลอีกต่อไป

**ผลกระทบจริงต่อ Friday launch: ต่ำ** — เพราะ LWW engine เป็น opt-in ผ่าน flag `lww_engine` ใน localStorage และ **default OFF** production ใช้ตัวคำนวณเก่า (chain-walk engine) เป็นหลักอยู่แล้ว ไม่ใช่ LWW — เรื่องนี้จึงเป็น "เอกสาร design เขียนไว้ว่าทำแล้วแต่จริงๆ ยังไม่ได้ทำ" มากกว่า "บั๊กที่กำลังกระทบคนใช้งานจริง"

**แนวทางแก้ที่แนะนำ:** **ไม่ต้องรีบก่อน Friday** — ตัดสินใจทีหลังระหว่าง (ก) เขียนสมุดบันทึกถาวรตามสเปคจริงๆ (งานใหญ่) หรือ (ข) แก้เอกสาร design ให้ตรงกับสิ่งที่โค้ดทำอยู่จริง (คำนวณสด ไม่ persist ก็เพียงพอสำหรับสเกลปัจจุบัน) — ไม่มีทางเลือกไหนที่ต้องรีบทำก่อนวันศุกร์
